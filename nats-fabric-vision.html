    How a message lives

    Core NATS, JetStream, KV & Object Store

    A message is published to a subject. Core NATS fans it out to every live subscriber instantly (at-most-once, microseconds). JetStream also captures it to a durable stream so it can be replayed, queued, or consumed later. KV and Object Store are streams in disguise — state and files that live on the bus.

    Core NATS

    Fire-and-forget pub/sub & request/reply. Microsecond latency, at-most-once. The subject is the only address. If nobody's listening, it's gone — that's a feature.

    JetStream

    Persistence on the same subjects. Streams replay history, work-queues guarantee exactly-once, RAFT replicates R=3. This replaces Kafka and Temporal.

    KV Store

    A stream as a key-value map. Revisions, TTL, optimistic locking, and watch — react the instant a value changes. Agent memory & node config live here.

    Object Store

    A stream as a filesystem. Chunked, versioned, watchable, 279 MiB/s. The whole Obsidian vault syncs here — any node pulls any file on demand.

    🔑

    The $SRV micro API is the daemon. Every service registers as a NATS micro service — discoverable via $SRV.PING, introspectable via $SRV.INFO, measured via $SRV.STATS. No service registry. No coordinator. No omd needed on the hub. The bus discovers itself.

    Polyglot by design

    Same wire protocol. Four languages. One job each.

    An agent is just a NATS service that answers prompts. Whether it speaks Go, Python, TypeScript, or Dart, nothing on the bus can tell — that's the entire point. Each language earns its place by doing the one thing it's best at. This is how the system scales to hundreds of nodes without becoming a monolith.

    The fleet-scale argument: 100 edge nodes = 100 tiny Go binaries (static, ARM, no deps). 15 hub services = one Python venv, one requirements.txt, one CI job.     Physical reality

    Homestead ⇄ Hub ⇄ Office ⇄ Cloud

    One logical bus, many physical locations. Each site runs a leaf node with embedded NATS — fully operational offline, then re-syncs to the hub when the link returns. Tailscale is the encrypted mesh underneath; the cloud VPS is the always-on public doorway. Drag-free, live — watch leaf traffic flow to the hub.

    Homestead — House · Barn · Greenhouse
    Hub — omos-mbp (standalone core)
    Office — workstation leaf
    Cloud — omos-cloud VPS (Tailscale Funnel)
    Mobile — Go2 · G1 · drone

    The hub stays standalone

    omos-mbp runs a plain nats-server. It never embeds — it's the anchor every leaf connects up to. JetStream streams (EVT/CMD/TEL/VOICE) live here, replicated.

    Leaves survive offline

    barn-pi loses wifi? Its embedded NATS keeps serving local sensors and the local agent. When the link returns, JetStream mirrors the gap. No data lost.

    Cloud is the doorway

    omos-cloud is a leaf with a public Tailscale Funnel entry. Voice from anywhere, webhooks from any SaaS, the app from your phone — all enter here, then ride the mesh home.

    SaaS, re-imagined

    No API calls in your code. Ever.

    Traditional access means writing HTTP clients, auth refresh, retry logic, and rate limiting forty times for forty providers. On the fabric, every SaaS provider is a subject pattern. An agent publishes a request; a Wombat connector or om-saas service does the HTTP; the response comes back as a subject and is captured in ClickHouse forever. Provider rotation becomes a config change.

    Webhooks are first-class

    No polling. Skool / GHL / Stripe fire → Wombat normalizes → bus event → subscribers react instantly. Inbound and outbound, same fabric.

    Full audit, free

    Every SaaS call flows through ClickHouse. Permanent trail: "show every GHL contact created in March" is one SQL query. Rate limits & retries live in the bridge.

    ⇄

    Why this beats API / MCP / CLI directly: those couple your agent to a vendor's shape and give you no record. Through the bus, the agent is decoupled (publish & await), the interaction is captured (ClickHouse), the provider is swappable (Wombat YAML), and any node on the fabric — even a robot or the voice agent — can trigger a SaaS action without knowing the API exists.
